  2. La Estrategia de Resiliencia: "Importante pero No Vital"

  Para que Rawrnot no muera si Google cambia su API o te cancela la cuenta, implementaremos la arquitectura de "Proveedores Intercambiables":

  A. El Patrón de Adaptador (Abstraction Layer)
  En el código de Swift, no escribiremos "Google Drive". Escribiremos un StorageProvider.
   * Rawrnot dirá: "Guarda este archivo".
   * El sistema buscará quién es el proveedor activo. Hoy es Google Drive. Mañana puede ser Amazon S3, Dropbox, un servidor propio (MinIO) o incluso el disco duro de tu Mac.
   * Resultado: Cambiar de Google a otro servicio sería cambiar una sola línea de configuración, no reprogramar el sistema.

  B. Principio de "Metadatos Primero"
  Rawrnot siempre guardará la información crítica en SU PROPIA base de datos (PostgreSQL).
   * Si guardamos un render, la base de datos de Rawrnot guarda: ID, Nombre, Cliente, Fecha.
   * El link de Google Drive es solo un "atributo extra".
   * Si Google muere: Rawrnot todavía sabe que el archivo existe y a quién le pertenece. Solo nos daría un error de "vínculo roto", pero la integridad de tu CRM y tu negocio sigue intacta. Podrías simplemente resubir los
     archivos a otro lado y el sistema seguiría funcionando.

  C. El "Buffer" Local
  Implementaremos una lógica de Cola de Sincronización:
   * Cuando un cliente sube algo, primero llega a Rawrnot (almacenamiento temporal). Luego, el sistema intenta subirlo a Drive "en segundo plano".
   * Si la API de Google está caída, el archivo se queda en el buffer de Rawrnot y lo reintenta más tarde. Tu usuario nunca ve el error.
